# MATCH POINT — API 연동 계약서 (백엔드/파트너용)

이 문서는 통합 점수 게시판이 각 앱(우리 **리매치** + 타 앱)으로부터 투표 데이터를 받아
합산하기 위해 필요한 **API 표준 스펙**입니다. 아직 공식 문서가 없어, 프론트가 기대하는
형태를 여기에 먼저 정의했습니다. **각 앱이 아래 표준을 따라주면 추가 매핑 없이 바로 연동**됩니다.
(앱마다 형태가 다르면, 서버리스 프록시 `api/scores.js` 의 `normalizeSource()` 에서 흡수합니다.)

---

## 1. 데이터 흐름 / 보안 구조

```
[브라우저(공개 링크)]  ──>  /api/scores (서버리스, 우리 도메인)  ──API Key──>  [각 앱 API]
        키 없음                  키는 환경변수로만 보관                       투표 원본 데이터
```

- **API Key 는 절대 프론트엔드에 두지 않습니다.** Vercel 환경변수로만 보관하고
  서버리스 프록시(`api/scores.js`)에서만 사용합니다.
- 프론트는 오직 같은 도메인의 `/api/scores` 만 호출합니다.

---

## 2. 각 앱 API 가 제공해야 하는 응답 (표준)

`GET {앱별 endpoint}`  — 인증: `X-API-Key: <key>` (헤더 이름은 협의 가능)

응답(JSON):

```json
{
  "items": [
    {
      "entryId": "t-01",           // 필수: 모든 앱에서 동일 항목은 같은 ID여야 합산됨
      "name": "후보 이름",          // 필수: 표시 이름 (후보/아티스트)
      "group": "트롯",             // 부문: "트롯" 또는 "KPOP" (사이트 상단 탭이 이 값으로 생성됨)
      "imageUrl": "https://.../t-01.jpg",  // 선택: 후보 사진(없으면 이름 첫 글자 아바타)
      "votes": 1280,               // 필수: 해당 앱에서의 투표수(0 이상 정수)
      "updatedAt": "2026-06-23T10:00:00+09:00"  // 선택: ISO 8601
    }
  ]
}
```

또는 최상위가 배열이어도 됩니다: `[ { "entryId": ... }, ... ]`

> **표시 정책**: 화면에는 **합산 총 투표수(`totalVotes`)만** 보여줍니다.
> 앱별 투표수(`votesBySource`)는 합산을 위해 내부적으로만 쓰이고 노출하지 않습니다.

### 필드 별칭 (자동 흡수)
프록시는 아래 별칭도 인식합니다 — 가능하면 위 표준 키를 권장.
- `entryId` ← `id`
- `name` ← `title`
- `votes` ← `count` / `score`

### ⚠️ 가장 중요한 규칙: **항목 ID 통일**
합산은 `entryId` 가 같은 항목끼리 이뤄집니다. 리매치의 "서울 다이노스"와
타 앱의 "서울 다이노스"가 합산되려면 **두 앱이 같은 `entryId`** 를 써야 합니다.
- 권장: 공통 ID 체계를 사전에 합의(예: `e-001`).
- 불가 시: 이름 기준 매칭 등 대체 매핑을 프록시에 추가해야 함(정확도 낮음, 비권장).

---

## 3. 통합 응답 (프록시 → 프론트, 참고용)

프론트가 받는 최종 형태입니다. (프론트는 이 형태만 의존)

```json
{
  "sources": [
    { "id": "rematch", "label": "리매치" },
    { "id": "partner-a", "label": "타 앱 A" }
  ],
  "entries": [
    {
      "id": "e-001",
      "name": "서울 다이노스",
      "group": "트롯",
      "votesBySource": { "rematch": 1280, "partner-a": 940 },
      "totalVotes": 2220,
      "updatedAt": "2026-06-23T10:00:00+09:00"
    }
  ],
  "fetchedAt": "2026-06-23T10:00:05+09:00"
}
```

---

## 4. 실연동 현황 (2026-06-26 기준)

표준 응답 협의 없이 **각 앱의 실제 응답 형태에 맞춰 프록시에서 흡수**하는 방식으로 연동했습니다.
설계/세부 스펙: `docs/superpowers/specs/2026-06-26-rematch-partner-integration-design.md`.

- 부문(투표) = 리매치 campaign ↔ 타앱 department 짝 (`api/scores.js` 의 `VOTES`):
  - `KPOP`(아이돌): 리매치 `campaignId=69d89a70820f8b2282f3fae6` + 타앱 `department_id=1`
  - `트롯`(대중가수): 리매치 `campaignId=69d89afc820f8b2282f3fbd1` + 타앱 `department_id=2`
- **명단·이름·사진의 원천 = 리매치 API** (`data.scoreDataList[].name_ko / portraitUrl`).
- **합산 키 = 후보 "이름" 자동 매칭** (NFC + 공백 제거 + 소문자 정규화 비교).
  - ⚠️ 두 앱의 표기가 다르면(예: 언어 차이) 매칭이 안 될 수 있음. 안 맞으면 그 후보는
    리매치 단독 점수만 표시됨. 정확도가 필요하면 명시적 ID 매핑표 도입을 검토.
- 인증: 양쪽 모두 헤더 `X-API-Key`.
- 부분 실패: 한쪽 출처만 실패하면 성공한 쪽 점수만으로 표시 (`Promise.allSettled`).
